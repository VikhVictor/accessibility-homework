# Accessibility Homework

Учебное Android-приложение для аудита доступности с TalkBack. Проект намеренно
содержит проблемы доступности: задача — найти их в реальном пользовательском
сценарии, описать влияние на пользователя и предложить корректное исправление.

## Две версии приложения

В проекте находятся две независимые реализации одних и тех же экранов и сценариев.
Их можно установить на одно устройство одновременно.

| Модуль | UI | Application ID |
| --- | --- | --- |
| `:app` | Android Views | `com.sweethome` |
| `:app-compose` | Jetpack Compose | `com.sweethome.compose` |
| `:data` | Общие данные и репозитории | — |

В обеих версиях доступны четыре экрана:

- каталог;
- информация о товаре;
- корзина;
- оформление заказа.

Поведение, данные и намеренные дефекты в версиях максимально сближены. Различия,
которые возникают из-за самого UI-фреймворка, нужно отдельно отметить в отчёте.

## Задача

1. Собрать и запустить обе версии приложения.
2. Включить TalkBack и пройти основной сценарий в каждой версии.
3. Найти и описать проблемы доступности.
4. Сравнить, как одна и та же проблема проявляется в Android Views и Compose.
5. Для каждой проблемы предложить исправление и способ повторной проверки.
6. Если в домашнее задание входит реализация исправлений — указать, что было
   изменено, и приложить результат повторного тестирования.

Нельзя ограничиваться проверкой XML, Compose-кода или Accessibility Scanner. В
отчёте должно быть фактическое поведение TalkBack на устройстве или эмуляторе.

## Сборка и запуск

Собрать обе версии:

```shell
./gradlew :app:assembleDebug :app-compose:assembleDebug
```

APK будут находиться здесь:

```text
app/build/outputs/apk/debug/app-debug.apk
app-compose/build/outputs/apk/debug/app-compose-debug.apk
```

Пример установки:

```shell
adb install -r app/build/outputs/apk/debug/app-debug.apk
adb install -r app-compose/build/outputs/apk/debug/app-compose-debug.apk
```

## Обязательный сценарий проверки

В каждой версии приложения нужно:

1. Открыть каталог и прослушать заголовок, категории и карточки товаров.
2. Перемещаться по экрану свайпами TalkBack и проверить горизонтальные списки.
3. Открыть товар, изучить описание и добавить товар в корзину.
4. Проверить изменение состояния корзины и перейти в неё.
5. Открыть оформление заказа.
6. Проверить выбор адреса, доставки и способа оплаты.
7. Активировать подтверждение заказа и проверить обратную связь.
8. Проверить возврат назад на каждом экране.

Дополнительные сценарии приветствуются, но этот маршрут обязателен. Иначе половина
дефектов просто мирно переживёт проверку.

## Что сдавать

Основной артефакт — `REPORT.md`, оформленный по
[готовому шаблону](REPORT_TEMPLATE.md).

В отчёте обязательны:

- устройство или эмулятор, версия Android и TalkBack;
- commit проекта, на котором выполнялась проверка;
- отдельные результаты для Android Views и Compose;
- шаги воспроизведения каждой проблемы;
- фактическое и ожидаемое поведение;
- влияние на пользователя и серьёзность проблемы;
- предлагаемое исправление;
- результат повторной проверки, если исправление реализовано;
- итоговое сравнение двух UI-реализаций.

Скриншот сам по себе не доказывает поведение TalkBack. Для спорных случаев нужен
короткий скринкаст либо текстовая расшифровка того, что озвучивает скринридер.

## Как учитывать проблемы

- Одинаковый дефект у повторяющихся карточек оформляется одной проблемой с указанием
  области распространения.
- Одинаковая проблема в Views и Compose оформляется одной карточкой отчёта, но с
  двумя отдельными наблюдениями.
- Если причина или поведение отличаются, это две разные проблемы.
- Функциональный баг без влияния на доступность отмечается отдельно и не заменяет
  accessibility-проблему.
- Количество строк не важнее качества воспроизведения. Выдумывать десятый дефект
  ради красивой цифры не требуется.

## Критерии оценки

| Критерий | Что проверяется |
| --- | --- |
| Полнота сценария | Пройдены обе версии и все четыре экрана |
| Воспроизводимость | Есть точные шаги и фактическое поведение TalkBack |
| Понимание влияния | Описано, что именно не может сделать пользователь |
| Корректность решения | Предложение исправляет причину, а не маскирует симптом |
| Сравнение реализаций | Показаны сходства и различия Views и Compose |
| Повторная проверка | Для реализованных исправлений есть результат ретеста |
